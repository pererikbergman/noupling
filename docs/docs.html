Installation

Homebrew (macOS / Linux)

brew tap pererikbergman/noupling
brew install noupling

From source

git clone https://github.com/pererikbergman/noupling.git
cd noupling
cargo install --path .

Prebuilt binaries

Download from GitHub Releases. Available for Linux (x86_64, aarch64), macOS (Apple Silicon, Intel), and Windows.

Quick Start

# 1. Scan your project
noupling scan /path/to/project

# 2. See the health score
noupling audit /path/to/project

# 3. Generate an interactive HTML report
noupling report /path/to/project --format html

# 4. Gate in CI
noupling scan . --diff-base origin/main
noupling audit . --fail-below 80

A .noupling/settings.json is auto-created on first run with sensible defaults. Edit it to customize thresholds, ignore patterns, and source extensions.

Health Score

The health score is a single number (0–100) that quantifies your architecture's coupling risk:

Score = 100 × (1 - TRI / (total_modules × max_weight))

TRI (Total Risk Index) is the sum of all violation RRIs. Each violation's RRI (Relationship Risk Index) is computed as direction_weight × density, where direction_weight reflects the severity of the dependency direction and density captures how many imports exist between the pair. See RRI and TRI for details.

RangeStatusAction
90–100HealthyMaintain
70–89WarningInvestigate new violations
0–69Needs attentionPrioritize decoupling

scan

Discover source files, parse imports, resolve dependencies, store in SQLite.

noupling scan <PATH> [--diff-base <BRANCH>]
FlagDescription
--diff-baseFilter results to files changed vs. this branch. Scans full project for resolution.

audit

Analyze the latest snapshot and display violations.

noupling audit [PATH] [--fail-below N] [--baseline] [--snapshot ID] [--module NAME]
FlagDescription
--fail-belowExit code 1 if score is below this. Use in CI.
--baselineOnly report violations not in the saved baseline.
--snapshotAudit a specific snapshot instead of the latest.
--moduleShow results for one module only (monorepo).

report

Generate a report file from the latest snapshot.

noupling report [PATH] --format <FORMAT> [--module NAME]

Formats: json, xml, md, html, sonar, mermaid, dot, bundle, dashboard, all. See Report Formats.

trend

Show health score history across snapshots.

noupling trend [PATH] [--last N]

baseline

Save current violations as accepted. Future --baseline audits only flag new ones.

noupling baseline save [PATH]
Pro-Tip: Commit .noupling/baseline.json to your repo. New team members inherit the accepted baseline and only see violations they introduce.

hook

Install a git pre-commit hook that fails on new violations.

noupling hook install [PATH]
noupling hook uninstall [PATH]

Bypass with git commit --no-verify.

Thresholds

{
  "thresholds": {
    "score_green": 90.0,
    "score_yellow": 70.0,
    "critical_severity": 0.5,
    "minimum_severity": 0.2,
    "hotspot_fan_in": 10,
    "min_cohesion": 0.1
  }
}

minimum_severity hides low-severity violations to reduce noise. critical_severity flags violations as critical in reports.

Risk Weights

Risk weights control how heavily each dependency direction contributes to the RRI score. Configure in .noupling/settings.json:

{
  "risk_weights": {
    "downward": 2,
    "sibling": 4,
    "upward": 6,
    "circular": 10
  }
}
DirectionWeightDescription
Downward2Dependency follows the declared layer order. Low risk, but still tracked.
Sibling4Dependency between modules at the same level. Creates horizontal coupling.
Upward6Dependency flows against the layer order. Inverts the architecture.
Circular10Part of a dependency cycle. Highest risk; prevents independent deployment.
Pro-Tip: Adjust weights to match your team's priorities. If upward dependencies are your biggest pain point, increase the upward weight to surface them more aggressively in the health score.

Coupling Mode

The coupling_mode setting controls which dependencies are treated as violations:

{
  "coupling_mode": "strict"
}
ModeBehavior
strict (default)All sibling coupling is a violation. Every cross-module dependency at the same level is flagged.
actionableOnly circular dependencies are violations. Sibling coupling is tracked but not penalized, reducing noise in projects where lateral dependencies are acceptable.
Guardrail: Start with strict mode on greenfield projects. Switch to actionable for legacy codebases where fixing all sibling coupling at once is impractical—focus on breaking cycles first.

Ignore Patterns

{
  "ignore_patterns": [
    "**/.git/**", "**/build/**", "**/node_modules/**",
    "**/generated/**", "**/__pycache__/**"
  ]
}

Glob patterns (gitignore-style). Any matching file is excluded from analysis.

Architectural Layers

Define layers so dependencies flowing downward (earlier → later) aren't flagged as coupling.

{
  "layers": [
    { "name": "presentation", "pattern": "**/ui/**" },
    { "name": "domain", "pattern": "**/domain/**" },
    { "name": "data", "pattern": "**/data/**" }
  ]
}
  • presentation → domain: allowed (downward)
  • data → presentation: violation (upward)
Guardrail: Layers suppress coupling violations but don't eliminate dependencies. If you find yourself adding many layers to make violations disappear, the real problem is your module boundaries.

Dependency Rules

{
  "dependency_rules": [
    {
      "from": "**/ui/**",
      "to": "**/data/**",
      "allow": false,
      "message": "UI must not depend on data layer directly"
    }
  ]
}

Inline Suppression

Suppress a specific import with a noupling:ignore comment on the same line or the line above:

import com.example.legacy.Helper // noupling:ignore

// noupling:ignore
import com.example.legacy.OtherHelper

Works with //, #, and -- comment styles. Disable globally with "allow_inline_suppression": false.

Pro-Tip: Treat noupling:ignore like @SuppressWarnings—use it sparingly for genuinely acceptable coupling, not to silence inconvenient truths.

Monorepo Support

Define independent modules, each analyzed separately with its own health score.

{
  "modules": [
    { "name": "app", "path": "app/src", "depends_on": ["lib-core"] },
    { "name": "lib-core", "path": "lib/core/src", "depends_on": [] }
  ]
}

depends_on declares the allowed cross-module import graph. Unlisted imports are flagged as cross-module violations.

# Multi-module summary
noupling audit .

# Single module detail
noupling audit . --module app

The overall score is a weighted average by file count. Files outside all modules are ignored. If modules is empty, the project is analyzed as a single module (backward compatible).

XS Metric

XS (Excess) quantifies the refactoring cost of violations.

  • Coupling violations: XS = number of import statements (the weight)
  • Circular dependencies: XS = break cost at the weakest link
  • Total XS: sum across all violations = total imports to remove

The weakest link is the hop in a cycle with the fewest imports—the cheapest edge to cut. The HTML report shows per-hop XS counts in expandable cycle details.

Diff Mode

noupling scan . --diff-base origin/main
noupling audit .

Scans the full project (needed for import resolution) but filters audit results to only violations touching changed files. Use in PRs to only fail on new issues.

RRI (Relationship Risk Index)

Each violation is assigned an RRI score that reflects both the severity of the dependency direction and the density of imports between the pair:

RRI = direction_weight × density

direction_weight comes from the risk weights configuration (e.g., 4 for sibling, 10 for circular). density is the number of import statements between the two modules.

Higher RRI means higher architectural risk. A single circular dependency with 3 imports scores 10 × 3 = 30, while a sibling coupling with 1 import scores only 4 × 1 = 4.

TRI (Total Risk Index)

The project-level risk metric, computed as the sum of all violation RRIs:

TRI = ∑ RRIi  (for all violations i)

Lower is better. TRI feeds directly into the health score formula. Track TRI over time with noupling trend to ensure architectural risk is decreasing.

Pro-Tip: TRI is additive, so you can compare modules directly: a module contributing 80% of the TRI is the obvious place to start refactoring.

Gravity Wells

A gravity well is a module whose total RRI exceeds 2× the median RRI across all modules. These are "God Objects" that pull the system into their orbit—everything depends on them, or they depend on everything.

Gravity wells are flagged in the audit output and highlighted in the dashboard. They represent the highest-leverage refactoring targets: reducing a gravity well's coupling has outsized impact on the overall health score.

Guardrail: A module can be a gravity well even with individually low-severity violations if it has many of them. Watch for modules that accumulate dozens of small couplings.

Red Flags

Red flags are specific anti-patterns detected by the risk framework:

Red FlagDetectionWhy it matters
Fused SiblingSibling coupling density > 2× median densityTwo modules so tightly coupled they are effectively one. Consider merging them or extracting a shared abstraction.
Trapped ChildModule has upward dependencies (child depends on parent layer)Inverts the dependency hierarchy. The "child" cannot be tested, deployed, or understood without its parent.
Pro-Tip: Red flags are intentionally opinionated. If a flagged pattern is correct for your architecture, suppress it with inline suppression or adjust your layer definitions.

Module Independence

For each top-level directory, the ratio of internal dependencies to total dependencies.

independence = internal_deps / (internal_deps + external_deps)

A module with independence 0.95 can be extracted into its own repo. One with 0.40 is deeply entangled. Modules below 70% are flagged in audit output.

Pro-Tip: Use this to answer "Can team X own this module independently?" High independence = ready to extract. Low independence = needs decoupling first.

Blast Radius

For each module, the count of other modules transitively affected if it changes. Computed via BFS on the reverse dependency graph.

If A imports B and B imports C, then C's blast radius is 2 (changing C affects both B and A). The top 10 highest blast radius modules are shown in audit output.

Use this before approving refactoring plans: low blast radius = safe to change, high = ripples through the codebase.

Instability (Martin's I)

Robert C. Martin's metric:

I = fan_out / (fan_in + fan_out)
  • I = 0.0: Maximally stable. Many depend on it; it depends on nothing. Should be abstract.
  • I = 1.0: Maximally unstable. Depends on many; nothing depends on it. Should be concrete.

The Zone of Pain: stable (I < 0.3) modules with high fan-in (≥ 5) are foundational but hard to change with wide impact. These are flagged separately.

Dependency Depth

The longest transitive dependency chain in the project, computed via topological sort on the dependency DAG.

Deep chains mean changes at the bottom ripple through many layers. Shallow chains mean the architecture is flat and manageable. The audit output shows the maximum depth and the critical path (the actual chain of files).

Violation Age

Violations are classified by how long they have existed across snapshots:

  • New: first appearance in the latest snapshot — urgent regressions
  • Recent: 2–4 snapshots — recent debt
  • Chronic: 5+ snapshots — accepted long-term debt

This helps prioritize: new regressions need immediate attention; chronic debt gets planned work.

  • Risk matrix: Instability vs Blast Radius scatter plot, color-coded
  • Architecture map: zoomable sunburst with dependency edges (gray) and violations (red)
  • Violation charts: by age, by type, plus quick-win refactoring targets
  • Dependency depth with critical path

Built for presenting to CTOs and tech leads. Self-contained HTML, dark or light theme, print-friendly.

Coupling Violations

A coupling violation occurs when sibling directories depend on each other. Each violation is scored using the RRI (Relationship Risk Index), which replaces the earlier depth-based severity model:

RRI = direction_weight × density

The direction_weight reflects the dependency direction (see Risk Weights) and density counts the import statements between the pair. Multiple imports between the same pair increase the RRI proportionally.

Pro-Tip: Not all coupling is bad. Use Layers to declare intended dependency directions so legitimate downward dependencies aren't flagged.

Circular Dependencies

A cycle exists when imports form a loop: A → B → C → A. Cycles are detected using Tarjan's SCC (Strongly Connected Components) algorithm to find all components, followed by mutual pair enumeration to surface the individual circular relationships within each SCC.

Each circular dependency receives the highest risk weight (default: 10) and its RRI scales with the density of imports in the cycle.

Cycles are grouped by order: 2 (mutual), 3 (triangular), N. Each includes the weakest link and break cost via the XS metric.

Hotspots & Cohesion

Hotspots are modules with high fan-in (many dependents). Changes to hotspots ripple widely. Default threshold: 10 incoming imports.

Cohesion measures how much files within a directory depend on each other:

cohesion = internal_deps / (file_count × (file_count - 1))

Low cohesion (< 0.1) suggests a directory groups unrelated code that might belong in separate packages.

Agentic Developer Experience

When using noupling with AI coding assistants, describe features in terms of intent rather than API:

  • "Check if my project has circular dependencies" → noupling scan . && noupling audit .
  • "Only fail the PR if I introduced new coupling" → noupling scan . --diff-base main && noupling audit . --fail-below 80
  • "Show me the easiest cycle to break" → look for the lowest break_cost in noupling audit output
  • "Suppress this known violation" → add // noupling:ignore on the import line
  • "This dependency is architecturally correct" → define it in layers or depends_on
Pro-Tip: Feed the JSON report to your AI assistant for codebase-aware architectural advice: noupling report . --format json, then share .noupling/report.json as context.
