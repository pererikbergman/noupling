Installation

Homebrew (macOS / Linux)

brew tap pererikbergman/noupling
brew install noupling

From source

git clone https://github.com/pererikbergman/noupling.git
cd noupling
cargo install --path .

Prebuilt binaries

Download from GitHub Releases. Available for Linux (x86_64, aarch64), macOS (Apple Silicon, Intel), and Windows.

Quick Start

# 1. Scan your project
noupling scan /path/to/project

# 2. See the health score
noupling audit /path/to/project

# 3. Generate an interactive HTML report
noupling report /path/to/project --format html

# 4. Gate in CI
noupling scan . --diff-base origin/main
noupling audit . --fail-below 80

A .noupling/settings.json is auto-created on first run with sensible defaults. Edit it to customize thresholds, ignore patterns, and source extensions.

Health Score

The health score is a single number (0–100) that quantifies your architecture's coupling:

Score = 100 × (1 - sum_of_severities / total_modules)
RangeStatusAction
90–100HealthyMaintain
70–89WarningInvestigate new violations
0–69Needs attentionPrioritize decoupling

scan

Discover source files, parse imports, resolve dependencies, store in SQLite.

noupling scan <PATH> [--diff-base <BRANCH>]
FlagDescription
--diff-baseFilter results to files changed vs. this branch. Scans full project for resolution.

audit

Analyze the latest snapshot and display violations.

noupling audit [PATH] [--fail-below N] [--baseline] [--snapshot ID] [--module NAME]
FlagDescription
--fail-belowExit code 1 if score is below this. Use in CI.
--baselineOnly report violations not in the saved baseline.
--snapshotAudit a specific snapshot instead of the latest.
--moduleShow results for one module only (monorepo).

report

Generate a report file from the latest snapshot.

noupling report [PATH] --format <FORMAT> [--module NAME]

Formats: json, xml, md, html, sonar, mermaid, dot, bundle. See Report Formats.

trend

Show health score history across snapshots.

noupling trend [PATH] [--last N]

baseline

Save current violations as accepted. Future --baseline audits only flag new ones.

noupling baseline save [PATH]
Pro-Tip: Commit .noupling/baseline.json to your repo. New team members inherit the accepted baseline and only see violations they introduce.

hook

Install a git pre-commit hook that fails on new violations.

noupling hook install [PATH]
noupling hook uninstall [PATH]

Bypass with git commit --no-verify.

Thresholds

{
  "thresholds": {
    "score_green": 90.0,
    "score_yellow": 70.0,
    "critical_severity": 0.5,
    "minimum_severity": 0.2,
    "hotspot_fan_in": 10,
    "min_cohesion": 0.1
  }
}

minimum_severity hides low-severity violations to reduce noise. critical_severity flags violations as critical in reports.

Ignore Patterns

{
  "ignore_patterns": [
    "**/.git/**", "**/build/**", "**/node_modules/**",
    "**/generated/**", "**/__pycache__/**"
  ]
}

Glob patterns (gitignore-style). Any matching file is excluded from analysis.

Architectural Layers

Define layers so dependencies flowing downward (earlier → later) aren't flagged as coupling.

{
  "layers": [
    { "name": "presentation", "pattern": "**/ui/**" },
    { "name": "domain", "pattern": "**/domain/**" },
    { "name": "data", "pattern": "**/data/**" }
  ]
}
  • presentation → domain: allowed (downward)
  • data → presentation: violation (upward)
Guardrail: Layers suppress coupling violations but don't eliminate dependencies. If you find yourself adding many layers to make violations disappear, the real problem is your module boundaries.

Dependency Rules

{
  "dependency_rules": [
    {
      "from": "**/ui/**",
      "to": "**/data/**",
      "allow": false,
      "message": "UI must not depend on data layer directly"
    }
  ]
}

Inline Suppression

Suppress a specific import with a noupling:ignore comment on the same line or the line above:

import com.example.legacy.Helper // noupling:ignore

// noupling:ignore
import com.example.legacy.OtherHelper

Works with //, #, and -- comment styles. Disable globally with "allow_inline_suppression": false.

Pro-Tip: Treat noupling:ignore like @SuppressWarnings—use it sparingly for genuinely acceptable coupling, not to silence inconvenient truths.

Monorepo Support

Define independent modules, each analyzed separately with its own health score.

{
  "modules": [
    { "name": "app", "path": "app/src", "depends_on": ["lib-core"] },
    { "name": "lib-core", "path": "lib/core/src", "depends_on": [] }
  ]
}

depends_on declares the allowed cross-module import graph. Unlisted imports are flagged as cross-module violations.

# Multi-module summary
noupling audit .

# Single module detail
noupling audit . --module app

The overall score is a weighted average by file count. Files outside all modules are ignored. If modules is empty, the project is analyzed as a single module (backward compatible).

XS Metric

XS (Excess) quantifies the refactoring cost of violations.

  • Coupling violations: XS = number of import statements (the weight)
  • Circular dependencies: XS = break cost at the weakest link
  • Total XS: sum across all violations = total imports to remove

The weakest link is the hop in a cycle with the fewest imports—the cheapest edge to cut. The HTML report shows per-hop XS counts in expandable cycle details.

Diff Mode

noupling scan . --diff-base origin/main
noupling audit .

Scans the full project (needed for import resolution) but filters audit results to only violations touching changed files. Use in PRs to only fail on new issues.

sonar.externalIssuesReportPaths=.noupling/noupling-sonar.json

Coupling Violations

A coupling violation occurs when sibling directories depend on each other. Severity is depth-weighted:

severity = weight / (depth + 1)

Root-level coupling (depth 0) is most severe. Multiple imports between the same pair are aggregated by weight.

Pro-Tip: Not all coupling is bad. Use Layers to declare intended dependency directions so legitimate downward dependencies aren't flagged.

Circular Dependencies

A cycle exists when imports form a loop: A → B → C → A. Detected via DFS on sibling adjacency graphs built from D_acc (accumulated external dependencies).

severity = total_modules / (depth + 1) / 10

Cycles are grouped by order: 2 (mutual), 3 (triangular), N. Each includes the weakest link and break cost via the XS metric.

Hotspots & Cohesion

Hotspots are modules with high fan-in (many dependents). Changes to hotspots ripple widely. Default threshold: 10 incoming imports.

Cohesion measures how much files within a directory depend on each other:

cohesion = internal_deps / (file_count × (file_count - 1))

Low cohesion (< 0.1) suggests a directory groups unrelated code that might belong in separate packages.

Agentic Developer Experience

When using noupling with AI coding assistants, describe features in terms of intent rather than API:

  • "Check if my project has circular dependencies" → noupling scan . && noupling audit .
  • "Only fail the PR if I introduced new coupling" → noupling scan . --diff-base main && noupling audit . --fail-below 80
  • "Show me the easiest cycle to break" → look for the lowest break_cost in noupling audit output
  • "Suppress this known violation" → add // noupling:ignore on the import line
  • "This dependency is architecturally correct" → define it in layers or depends_on
Pro-Tip: Feed the JSON report to your AI assistant for codebase-aware architectural advice: noupling report . --format json, then share .noupling/report.json as context.
