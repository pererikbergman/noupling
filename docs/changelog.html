Changelog

Not a commit log. A narrative of architectural decisions—what broke, what was built, and why each change exists.

v0.4.1 — The Visualization Release

April 2026

feat Sunburst dependency visualization (#123)

The friction: Existing reports showed violations as lists and tables. Teams couldn't see the overall shape of their architecture—where coupling clusters lived, how dependencies flowed between layers, or which areas were cleanly separated.

The implementation: A zoomable D3.js sunburst where concentric rings represent the directory hierarchy, color-coded by health score. Curved dependency edges connect inner ring segments, aggregated from all children underneath (using D_acc). Click any segment to zoom in with a smooth 750ms animated transition. Click the center to zoom back out. Edge hover tooltips show the exact from/to connection and import count.

The ripple effect: Architecture becomes visible at a glance. Dense edge bundles between two areas immediately reveal coupling hotspots. The drill-down lets teams explore from the 10,000-foot view down to individual directory pairs.

v0.4.0 — The Architecture-Aware Release

April 2026

feat Monorepo support (#84)

The friction: Large codebases with multiple modules (app, lib-core, lib-network) were analyzed as a single blob. A coupling violation in lib-core dragged down the score for the entire repo, making the number meaningless for teams that owned individual modules.

The implementation: Each module defined in settings.json gets independent D_acc computation, BFS coupling detection, and cycle finding. Cross-module imports are validated against a declared depends_on graph. The overall score is a weighted average by file count.

The ripple effect: Teams can now own their module's health score independently. The --module flag lets CI pipelines gate on individual modules, and cross-module violations make implicit dependencies explicit.

feat Inline suppression (#73)

The friction: Some coupling is intentional. A data layer importing a domain entity for mapping is architecturally correct in many patterns, but noupling flagged it. Teams had no escape hatch short of disabling the tool.

The implementation: // noupling:ignore on the import line (or the line above) excludes that dependency from analysis. Works with //, #, and -- comment styles. The suppressed count is tracked and reported so suppressions don't become invisible.

The ripple effect: Teams can adopt noupling incrementally without false-positive friction, while the suppressed count prevents noupling:ignore from becoming the new @SuppressWarnings("all").

feat XS metric — weakest link analysis (#112)

The friction: Circular dependencies were detected and scored, but the output didn't answer the practical question: "which import should I remove first?" Teams saw a cycle of order 5 and didn't know where to start.

The implementation: For each circular dependency, track import counts per hop. The hop with the fewest imports is the weakest link—the cheapest edge to cut. total_xs sums all imports needing removal across all violations.

The ripple effect: Refactoring is now prioritizable. A cycle with break cost 2 is a 10-minute fix. One with break cost 128 is a planned effort. The HTML report shows per-hop XS counts in expandable details.

feat Architectural layer enforcement (#92, #110)

The friction: Clean architecture projects (presentation → domain → data) had dozens of "violations" that were actually correct downward dependencies. The noise made real violations invisible.

The implementation: Define ordered layers in settings.json. Dependencies flowing downward are suppressed from coupling violations. Upward dependencies (data → presentation) are flagged as both coupling and layer violations.

feat Custom dependency rules (#91)

Glob-based allow/forbid rules for specific import patterns. Enforce boundaries like "UI must not import data layer directly" with custom messages.

feat Per-directory cohesion scoring (#64)

Measures how much files within a directory depend on each other. Low cohesion flags directories that group unrelated code—candidates for splitting.

Also in this release:

  • feat Dart, PHP, and Ruby parsers — 14 languages supported (#116, #86, #85)
  • feat GitHub PR comment bot workflow (#80)
  • feat Trend command for score history (#66)
  • feat Version string in all reports (#101)

v0.2.0 — The Multi-Language Release

April 2026

feat 11 language parsers

The friction: The initial release only supported Rust. Most real-world projects are polyglot, especially Android (Kotlin + Java) and web (TypeScript + JavaScript).

The implementation: Each language gets a Tree-sitter parser that extracts import statements from the AST and a resolver that maps imports to project-relative file paths. Parallel scanning via Rayon keeps performance constant regardless of language count.

feat Interactive HTML report

Kover-style drill-down navigation with color-coded scores per directory, breadcrumbs, and grouped circular dependencies by cycle order.

feat Diff mode for PR gating

The --diff-base flag scans the full project but filters results to changed files. CI pipelines can fail PRs on new violations without blocking on existing debt.

Also in this release:

  • feat 7 report formats: JSON, XML, Markdown, HTML, SonarCloud, Mermaid, DOT
  • feat Baseline files for incremental adoption
  • feat Hotspot detection (fan-in/fan-out)
  • feat Git pre-commit hook
  • feat Homebrew tap with auto-update on release

v0.1.0 — The Foundation

April 2026

feat Core architecture

The problem: Most linters check code style. Nothing checks whether your modules actually maintain the boundaries you drew on the whiteboard.

The approach: Four-stage pipeline: scan (discover + parse via Tree-sitter), store (SQLite snapshots), analyze (D_acc + BFS + DFS cycle detection), report.

Key algorithm: D_acc (accumulated dependencies) is computed bottom-up: for each directory, union all external dependencies from its subtree. BFS walks the tree top-down checking sibling pairs. Circular dependencies are found via DFS on sibling adjacency graphs.

  • Rust parser with Tree-sitter
  • SQLite storage with snapshot history
  • Severity scoring: weight / (depth + 1)
  • Health score: 100 × (1 - sum / total)
  • Configurable settings with auto-creation
  • Cross-platform CI (Linux, macOS, Windows)
